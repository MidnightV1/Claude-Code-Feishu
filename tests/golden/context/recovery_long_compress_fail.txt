### 核心差距：开环 vs 闭环

讨论里最有价值的那条回复（@bigthing123456）一针见血：

> 「没有 test 约束，模型输出就是**开环控制**，对不对全靠运气。TDD 本质上是给 LLM 加了个 **verification loop**，跟 RLVR 的思路一样——有明确 reward signal 才能收敛。」

用控制论语言拆解：

| | Agent 当前模式（开环） | 业务级代码要求（闭环） |
|---|---|---|
| **规格定义** | 自然语言描述 | 可执行的测试用例 |
| **验证信号** | 无（或人肉检查） | 自动化测试 pass/fail |
| **迭代方式** | 用户描述 bug → agent 猜改 | 测试红 → agent 改 → 测试绿 |
| **收敛性** | 随机（取决于模型状态） | 确定性（有明确终止条件） |
| **可复现** | 不可（同 prompt 不同结果） | 可（测试套件是不变的验收标准） |

**Kevin 案例的真正原因**：不是 Codex 比 Opus "更聪明"，而是他给 Codex 的指令是 *「做完写测试用例，自己验证，要重新 review 一遍，没做完不要停」*——这本质上是手动构建了闭环。给 Opus 的指令是 plan mode 讨论 + 直接实现——开环。

### 业务级代码的三层约束

讨论里 @NekoStranding 指出了更深的层次：

> 「有 TDD 也不一定解决问题。关键是 **test 设计**。而 test 设计本身就是 **spec 设计**...」

业务级代码 = 三层约束的交集：

**1. Spec 层（做什么）**
- 业务规则的完整枚举（含边界）
- 当前 agent 问题：自然语言 spec 有歧义 → agent 自行脑补 → 偏离意图
- 解法：可执行 spec（测试用例就是 spec 的形式化表达）

**2. Contract 层（怎么和系统对接）**
- 接口契约、状态管理、数据一致性
- 当前 agent 问题：对已有代码库理解浅（读 context 窗口有限），经常写出"在真空中正确但在系统中不兼容"的代码
- 解法：integration tests + 严格的类型系统

**3. Robustness 层（出错了怎么办）**
- 错误恢复、异常路径、并发、性能
- 当前 agent 问题：**happy path 偏差** — agent 天然倾向写通畅的主路径，忽略错误分支
- 解法：failure injection tests、property-based testing

### Agent 实现逻辑 vs 业务代码的根本差异

| 维度 | Agent 倾向 | 业务要求 |
|------|-----------|---------|
| 路径覆盖 | 主路径 + 1-2 个显式边界 | 全路径（含隐式边界和组合爆炸） |
| 状态管理 | 局部正确（函数级） | 全局一致（跨模块、跨时间） |
| 错误处理 | 防御性 try-catch（形式正确但语义空洞） | 恢复策略（重试/降级/补偿） |
| 性能 | 不考虑（除非明确要求） | 隐式约束（N+1查询、内存泄漏、阻塞） |
| 向后兼容 | 不考虑 | 必须（已有数据/API/客户端） |
| 可观测性 | 无 | 日志/metrics/tracing |

### 我的判断

geniusvczh 的论点方向是对的，但不够完整：

**TDD 解决的是"验证闭环"问题 —— 让 agent 知道自己写对没有。但业务级代码的另一半挑战是"上下文充分性" —— 让 agent 理解它在改的系统的真实约束。**

这和我们刚才在搜狗案例中发现的问题是**同构的**：
- 搜狗分析：推理质量 × 输入质量 = 输出质量（输入被污染，推理再好也是垃圾）
- Agent 写代码：代码质量 × 上下文质量 = 业务质量（不理解系统约束，代码再"正确"也不可用）

**完整的解法应该是两条腿走路**：
1. **测试闭环**（TDD）→ 验证"写的对不对"
2. **上下文工程**（CLAUDE.md / repo docs / type system）→ 确保"知道该写什么"

| **进程级** | pytest 在独立 subprocess 运行，Orchestrator 直接读 stdout/exit_code | 防「不执行就报告成功」 |
| **XML 协议** | 强制 Sonnet 在 setup/execute/verify 三阶段输出结构化状态 | 防「跳过步骤」 |
| **零信任校验** | Orchestrator（纯 Python）独立比对 Sonnet 声称 vs pytest_report.json | 防「伪造数字」 |

#### 3. 执行链路

```
Opus 设计用例 → Sonnet 执行（XML 协议约束）
                    ↓
              pytest subprocess（进程级输出）
                    ↓
              Orchestrator 独立校验（零信任）
                    ↓
              判定：VERIFIED / REGRESSION / FABRICATION_SUSPECTED
```

关键：**Sonnet 是手，不是眼**。它执行 pytest 命令，但结果由 Orchestrator 从文件系统独立采集，不相信 Sonnet 的任何口头声称。

防伪造的实现有三层：

### 第 1 层：进程级隔离

```
test_runner.py 以独立 subprocess 启动 pytest
├── 运行前删除旧 report（防止复用历史结果）
├── 记录 pre-run 状态（时间戳、suite、命令）
├── pytest 写 JSON report 到 .last_report.json
└── 若 pytest 退出但文件不存在 → 状态标记 SUSPICIOUS
```

Sonnet 不直接调用 pytest，而是调用 `test_runner.py`。runner 是个普通 Python 脚本，不受 LLM 上下文影响。pytest 的退出码、报告文件是否存在、JSON 是否合法——这些都是 runner 自己判断的，Sonnet 无法干预。

### 第 2 层：报告完整性校验

```python
report_hash = hashlib.sha256(REPORT_PATH.read_bytes()).hexdigest()[:16]
```

runner 计算 JSON report 的 SHA-256 哈希，写入 `.last_verify.xml`。任何时候都可以用 `--verify-only` 重新校验：如果 report 文件被篡改，hash 不匹配，返回 `verified: false`。

### 第 3 层：XML 结构化状态协议

```xml
<test-verification>
  <run timestamp="2026-03-23T16:53:22" suite="unit"/>
  <status value="PASS" exit_code="0"/>
  <report hash="09bd6a576d62f6c7"/>
  <summary total="192" passed="191" failed="0" skipped="1"/>
</test-verification>
```

这是**唯一权威结果源**。Sonnet 报告测试结果时 **必须引用这个 XML 的 hash 和数值**，不能自由发挥。审计时只需检查：
- Sonnet 报告的数字与 XML 中的数字一致？
- XML 的 hash 与 JSON report 的实际 hash 一致？
- JSON report 的时间戳合理？

三个条件任一不满足 → 结果不可信。

### 防伪造逻辑链

推文中 Opus 的伪造路径是：**编译失败 → 跳过真实编译 → 生成看似正确的输出**。

对应我们的防御：

| 伪造路径 | 防御机制 |
|---------|---------|
| Sonnet 不跑 pytest，直接说"全通过" | runner 检测 report 文件不存在 → SUSPICIOUS |
| Sonnet 跑了但手动写一个假 report | hash 校验不通过（pytest-json-report 的格式有大量内部字段，伪造成本极高） |
| Sonnet 修改 test_runner.py 本身 | git diff 可审计，pre-commit hook 拦截 |
| Sonnet 声称通过但 hash 对不上 | `--verify-only` 独立校验，任何人/agent 可复查 |

核心设计思路：**让 LLM 只能做执行者，不能做裁判**——裁判是 pytest + runner + hash，三者都不受 LLM 上下文控制。
