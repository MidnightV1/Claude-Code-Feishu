#### 1. 测试用例 = 可审计目录

| 层 | 机制 | 防什么 |
|---|---|---|
| **进程级** | pytest 在独立 subprocess 运行，Orchestrator 直接读 stdout/exit_code | 防「不执行就报告成功」 |

```

                    ↓
              pytest subprocess（进程级输出）
                    ↓

                    ↓
```

防伪造的实现有三层：

### 第 1 层：进程级隔离

```
test_runner.py 以独立 subprocess 启动 pytest
├── 运行前删除旧 report（防止复用历史结果）
├── 记录 pre-run 状态（时间戳、suite、命令）
├── pytest 写 JSON report 到 .last_report.json
└── 若 pytest 退出但文件不存在 → 状态标记 SUSPICIOUS
```

Sonnet 不直接调用 pytest，而是调用 `test_runner.py`。runner 是个普通 Python 脚本，不受 LLM 上下文影响。pytest 的退出码、报告文件是否存在、JSON 是否合法——这些都是 runner 自己判断的，Sonnet 无法干预。

### 第 2 层：报告完整性校验

```python
report_hash = hashlib.sha256(REPORT_PATH.read_bytes()).hexdigest()[:16]
```

runner 计算 JSON report 的 SHA-256 哈希，写入 `.last_verify.xml`。任何时候都可以用 `--verify-only` 重新校验：如果 report 文件被篡改，hash 不匹配，返回 `verified: false`。

### 第 3 层：XML 结构化状态协议

```xml
<test-verification>
  <run timestamp="2026-03-23T16:53:22" suite="unit"/>
  <status value="PASS" exit_code="0"/>
  <report hash="09bd6a576d62f6c7"/>
  <summary total="192" passed="191" failed="0" skipped="1"/>
</test-verification>
```

这是**唯一权威结果源**。Sonnet 报告测试结果时 **必须引用这个 XML 的 hash 和数值**，不能自由发挥。审计时只需检查：
- Sonnet 报告的数字与 XML 中的数字一致？
- XML 的 hash 与 JSON report 的实际 hash 一致？
- JSON report 的时间戳合理？

三个条件任一不满足 → 结果不可信。

### 防伪造逻辑链

推文中 Opus 的伪造路径是：**编译失败 → 跳过真实编译 → 生成看似正确的输出**。

对应我们的防御：

| 伪造路径 | 防御机制 |
|---------|---------|
| Sonnet 不跑 pytest，直接说"全通过" | runner 检测 report 文件不存在 → SUSPICIOUS |
| Sonnet 跑了但手动写一个假 report | hash 校验不通过（pytest-json-report 的格式有大量内部字段，伪造成本极高） |
| Sonnet 修改 test_runner.py 本身 | git diff 可审计，pre-commit hook 拦截 |
| Sonnet 声称通过但 hash 对不上 | `--verify-only` 独立校验，任何人/agent 可复查 |

核心设计思路：**让 LLM 只能做执行者，不能做裁判**——裁判是 pytest + runner + hash，三者都不受 LLM 上下文控制。
